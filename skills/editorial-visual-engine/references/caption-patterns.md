# Caption Pattern Library

This library captures reusable structures from the permanent examples in `caption-reference-corpus.md`.

The corpus is a reference for pacing, information order, list rhythm, proof placement, and CTA shape. It is not permission to copy a creator's wording, identity, claims, links, or personal story.

## Priority

Use caption inputs in this order:

1. `voice.md` and `about-me.md`, when available
2. User-supplied caption references for the current run
3. `caption-reference-corpus.md`
4. The structural rules in this file

A voice profile controls tone and rhythm. The corpus controls structure only.

## Reference selection method

Before drafting, score the current source against the reference corpus:

- **Content shape:** list, workflow, announcement, argument, migration, guide, collection, or product demo
- **Evidence shape:** source facts, personal experience, visible output, time condition, or inventory
- **Reader job:** learn, decide, install, save, comment, visit, or understand
- **Density:** short thesis, medium explanation, or detailed numbered breakdown
- **CTA need:** none, question, source visit, save, repost, comment keyword, or install action

Select one primary reference. A second reference may supply one mechanism only, such as a list rhythm or CTA shape.

## Pattern 1. Time-sensitive field guide

**Corpus reference:** Reference 01

Best for free windows, release deadlines, access changes, and temporary availability.

Structure:

1. State the time-sensitive change immediately
2. Explain the deadline or access condition
3. List 3 to 7 practical things to do before the change
4. Add one useful prompt or action under each item
5. Restate the deadline
6. Close with one natural question or sharing cue

Avoid false urgency. The source must support the date and condition.

## Pattern 2. Build decision and operational lesson

**Corpus reference:** Reference 02

Best for migrations, process changes, design-to-code stories, and internal workflow changes.

Structure:

1. Open with the decisive action
2. Admit the cost or difficulty
3. Explain what the old process prevented
4. Show the new behaviour across several roles
5. State the compounding lesson
6. End without a forced CTA when the lesson is strong enough

## Pattern 3. Launch with hidden complexity

**Corpus reference:** Reference 03

Best for skills, tools, templates, creative workflows, and product launches.

Structure:

1. Announce what is now possible
2. Contrast the simple output with the work behind it
3. Name the operational pain
4. Explain the mechanism or stages
5. Clarify who it is for
6. Explain what is included
7. Give the access instruction

Do not claim performance readiness or business results unless the source supports them.

## Pattern 4. Complete workflow inventory

**Corpus reference:** Reference 04

Best for repositories, GTM workflows, automation chains, and operational toolkits.

Structure:

1. State the end-to-end outcome
2. Define the start and finish of the workflow
3. List the components with one practical explanation each
4. Name the handoff between components
5. Summarise what the reader receives
6. Add a simple access CTA when relevant

## Pattern 5. Save-before-you-start guide

**Corpus reference:** Reference 05

Best for setup instructions, preflight workflows, and build sequences.

Structure:

1. Tell the reader to save the setup before beginning
2. Present numbered steps in direct language
3. Include one exact command or prompt where useful
4. Remove unnecessary theory
5. State the final outcome and realistic time only when supported
6. Explain where to get the full resource

## Pattern 6. Advisor-board concept demo

**Corpus reference:** Reference 06

Best for agent roles, panels, simulations, personas, and decision-support skills.

Structure:

1. Open with a surprising hypothetical
2. Confirm that it is possible
3. Give the shortest installation path
4. Demonstrate several distinct viewpoints
5. Show how the viewpoints are synthesised
6. End with a question that helps readers imagine their own version

## Pattern 7. Blocked capability and access guide

**Corpus reference:** Reference 07

Best for API access, approval processes, account restrictions, setup blockers, and compliance steps.

Structure:

1. State the tool benefit
2. Reveal the blocker
3. Explain what remains impossible while blocked
4. Name the required access level or change
5. Explain the rejection or failure risk
6. List what the guide contains
7. Give one clear access CTA

Do not turn an unofficial workaround into an endorsed platform rule.

## Pattern 8. Conceptual stack argument

**Corpus reference:** Reference 08

Best for strategy, marketing systems, role-change commentary, and connected operating models.

Structure:

1. Name the common simplification
2. Show the visible layer
3. Reveal the hidden connected layers
4. Explain how one weak link harms the rest
5. Reframe the concept as a connected operating model
6. Close with the core thesis

## Pattern 9. Product demo with output proof

**Corpus reference:** Reference 09

Best for working tools, repositories, demos, automation products, and before-versus-after output.

Structure:

1. State the result in the first line
2. Clarify that the shown output is real
3. Quantify the input and output when supported
4. Remove the old costs or dependencies
5. Explain the mechanism at a useful level
6. Point to proof inside the post or source
7. Give the shortest path to access

Do not add proof metrics from the reference to another product.

## Pattern 10. Scheduled workflow list

**Corpus reference:** Reference 10

Best for recurring agents, monitoring loops, scheduled reports, and operational automation.

Structure:

1. State the number of workflows and the job they collectively perform
2. Add one short line explaining the operating rhythm
3. Number each workflow
4. Under each item, explain the action and the practical outcome
5. End with a role-change or operating-model insight
6. State setup effort only when the source supports it
7. Add one access CTA

## Pattern 11. Collection organised as a company

**Corpus reference:** Reference 11

Best for large skill libraries, tool directories, department maps, and category-led collections.

Structure:

1. Open with the total and organising metaphor
2. Group the collection into clear departments or categories
3. List representative items under each group
4. Summarise the scope of each category
5. Confirm whether every listed item is real and available only when verified
6. Close with a sharing or source CTA

## LinkedIn rules

- Normally 150 to 300 words, but allow longer inventory posts when the source needs it
- Use short paragraphs for feed reading
- The caption should add context beyond the graphic
- Keep numbers, names, dates, and access conditions identical to the source
- Use one primary CTA, not several competing CTAs
- Do not add hashtags unless requested
- Do not import personal experience from a reference caption
- Do not use a comment-keyword CTA unless the current user can fulfil it

## X rules

Use one post when the core idea and evidence fit without compression damage.

Use a thread when the source contains:

- More than four distinct steps
- A workflow with dependencies
- Several verified examples
- A before and after story that needs context
- A category inventory that cannot fit cleanly in one post

The first post must stand alone. Later posts should add information rather than restate the hook.

## Caption alignment check

Before returning the caption, confirm:

- The selected angle matches the visual angle
- The first line does not promise more than the source contains
- Every metric and time condition comes from the current source
- The caption adds useful context instead of transcribing the image
- The CTA is possible for the current user to fulfil
- The final copy follows `voice.md` when available
- No sentence was copied from the corpus unless it is a generic command or the user explicitly supplied that sentence for reuse
